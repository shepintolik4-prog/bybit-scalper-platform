# Архитектура институциональной торговой платформы

Документ задаёт **целевое состояние** системы в `bybit-scalper-platform/` и границы ответственности модулей. Реализация идёт итерациями; уже существующие сервисы (regime, meta-filter, execution, risk 1.0, backtest) остаются **совместимыми слоями**.

## 2. Портфельное управление (текущий этап)

**Модуль:** `app/services/portfolio_manager.py`

- **Входы:** ковариационная матрица доходностей (или прокси), ожидаемые доходности (опционально), ограничения (веса, корреляции), режим/сигналы для динамики.
- **Выходы:** вектор весов `w` (сумма = 1), разбиение капитала по активам в USDT, диагностика (метод, итерации, штрафы за ограничения).
- **Методы:**
  - **Risk parity** (обратная волатильность; опционально итеративный ERC).
  - **Mean-variance** (квадратичная оптимизация под `scipy`, ограничения box + сумма = 1).
  - **Ограничение корреляций:** сжатие матрицы (Ledoit–Wolf‑лайт), проекция весов при чрезмерной совместной экспозиции.
  - **Динамическая аллокация:** смешивание базовых весов с вектором силы сигналов / режима (без нарушения суммарных лимитов).

**Интеграция с ботом (следующие шаги):** `bot_engine` перед открытием позиции запрашивает «бюджет» по символу из `PortfolioManager`; риск‑движок проверяет портфельный DD и VaR (Risk Engine 2.0).

## 3. Research pipeline (план)

- **Feature store:** версионированные таблицы/файлы признаков по `(symbol, ts, feature_set_version)`.
- **Эксперименты:** `experiments/` или MLflow-совместимые JSON-артефакты + метрики в PostgreSQL.
- **Сравнение стратегий:** единый отчёт по walk-forward, realistic backtest, стабильности по режимам.

## 4. Execution engine (расширение)

- TWAP/VWAP **симуляция** в бэктесте; live — постановка лимитов/айсбергов (зависит от биржи).
- Проверка **ликвидности** (объём стакана / ADV) перед крупным ордером.
- Жёсткие **лимиты проскальзывания** относительно mid.

## 5. Meta-strategy layer

- Реестр стратегий: `scalp`, `trend` с разными порогами и размером.
- **Роутинг:** по `MarketRegime` выбирается вес стратегии или полное переключение.

## 6. Risk Engine 2.0

- Портфельный DD, **VaR/CVaR** (исторический или параметрический), сценарные стресс-тесты.
- Связь с `portfolio_manager` и глобальным kill-switch.

## 6b. Quant-fund / PM (концентрация, ERC-выбор)

- Реализовано в `app/services/fund_risk.py`: потолок доли крупнейшего имени в gross notional, опционально — доля альт-кластера vs мейджоры, vol-targeting по истории эквити, при нескольких кандидатах за тик — **ERC + signal tilt** вместо простого `argmax |edge|`.
- Документация и env: **`docs/QUANT_FUND_RISK.md`**, префикс `FUND_*`. API: `GET /api/risk/fund`.

## 7. Наблюдаемость и алерты

- **Sentry:** исключения, необработанные ошибки API, опционально «business» события (failed order).
- **Datadog:** метрики HTTP (DogStatsD из `app/monitoring/`), агент — локально `docker-compose.datadog.yml` или sidecar на ECS; см. **`docs/MONITORING.md`**.
- Алерты: порог DD, остановка торговли, рост ошибок 5xx.

## 8. Инфраструктура

- **Текущее:** Docker Compose (Postgres + API + UI).
- **AWS:** образы в ECR, ECS Fargate, RDS Postgres, Secrets Manager, ALB — черновик в **`infra/aws/README.md`** (`infra/aws/ecs/task-definition.json`, Terraform-каркас).
- **Масштабирование воркеров:** отдельный процесс/контейнер `worker` с тем же кодом, флаг `ROLE=worker`, очередь из Redis/SQS (этап 2).

## 9. Интеграции плагинов Cursor (политика репозитория)

| Сервис    | Роль в репо                                      |
|-----------|--------------------------------------------------|
| Sentry    | Уже в коде; DSN через env                        |
| Datadog   | DogStatsD + env `METRICS_*`/`DD_*`; см. `docs/MONITORING.md` |
| AWS       | `infra/aws/README.md`, `infra/aws/ecs/task-definition.json`   |
| Vercel    | Фронт: `frontend`, `VITE_API_BASE`               |
| Firecrawl | `app/services/news_ingest_stub.py` + README    |
| Browserbase | опционально для headless; не обязателен        |
| Stripe    | `docs/SAAS_STRIPE.md` — webhooks, customer       |

Код не должен жёстко импортировать закрытые SDK там, где достаточно env и HTTP — это упрощает тесты и CI.
